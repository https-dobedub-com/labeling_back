# labeling_back 작업 계획서 규칙

이 문서는 `plan/` 폴더에 작성하는 작업 계획서의 형식과 운영 규칙을 정의한다.

## 1. 기본 원칙

- 모든 작업을 시작하기 전에 `plan/` 폴더에 계획서를 먼저 작성한다.
- 계획서 없이 코드 작업을 시작하지 않는다.
- **계획서를 작성한 후 반드시 작업을 멈추고 사용자의 승인을 기다린다.**
- 사용자가 승인하면 그때 작업을 시작한다.
- 승인 전에 코드 수정, 파일 생성 등 실제 구현 작업을 절대 진행하지 않는다.

## 2. 폴더 구조 — 도메인별 분류

```text
plan/
  clip/
    20260324_1010_clip-domain-plan.md
    20260324_1430_clip-query-plan.md
  etc/
    20260319_0900_env-config-plan.md
```

- 도메인 이름으로 하위 폴더를 만들고, 해당 도메인의 계획서를 그 안에 둔다.
- 도메인 분류가 애매한 작업은 `etc/` 폴더에 넣는다.
- 여러 도메인에 걸친 작업은 **주 도메인** 폴더에 넣거나, 명확하지 않으면 `etc/`에 넣는다.

## 3. 파일명 규칙

```
<YYYYMMDD>_<HHmm>_<작업명>-plan.md
```

- **날짜와 시간을 파일명 맨 앞에** 둔다 — 파일시스템에서 시간순 정렬이 보장된다.
- 날짜(`YYYYMMDD`)와 시간(`HHmm`, 24시간제) 사이를 `_`로 구분한다.
- 시간 뒤 `_`로 구분하여 작업명을 kebab-case로 작성한다.
- 예시: `20260324_1010_clip-domain-plan.md`, `20260324_1430_clip-query-plan.md`

## 4. 계획서 템플릿

```markdown
# <작업 제목>

- 대상 경로: `<repo-relative path 또는 domain>`

## 개요
- 작업 목적: 왜 이 작업을 하는지 (배경, 요구사항)
- 작업 범위: 이번 작업에서 다루는 범위와 다루지 않는 범위

## 현재 상태 분석
- 관련 파일의 현재 구조와 상태를 정리한다.
- 기존 코드에서 변경이 필요한 부분을 명시한다.
- 관련 도메인의 context 문서 내용을 확인하고 반영한다.

## 변경 계획

### 생성할 파일
| 파일 경로 | 역할 | 비고 |
|-----------|------|------|
| `src/services/xxx/...` | 설명 | |

### 수정할 파일
| 파일 경로 | 변경 내용 | 비고 |
|-----------|-----------|------|
| `src/services/xxx/...` | 설명 | |

### 삭제할 파일
| 파일 경로 | 삭제 이유 |
|-----------|-----------|
| (없으면 생략) | |

## 구현 상세

### 1. <작업 단위 제목>
- 구현할 내용을 구체적으로 기술한다.
- 필요시 코드 스니펫, 시그니처, 타입 정의 등을 포함한다.

### 2. <작업 단위 제목>
- ...

## 체크리스트
- [ ] 첫 번째 작업
- [ ] 두 번째 작업
- [ ] 세 번째 작업
- [ ] build 검증

## 참고사항
- 참조한 rules 문서 목록
- 참조한 context 문서 목록
- 주의사항, 제약조건, 의존성

## 구현 결과 요약
- (모든 체크리스트 완료 후 작성)
- 생성/수정/삭제된 파일 목록
- 주요 변경 사항 요약
- 특이사항이나 후속 작업이 필요한 부분
```

### 4-1. 템플릿 작성 규칙

- **개요**: "왜 하는지"와 "범위"를 명확히 구분한다. 범위 밖의 작업은 명시적으로 제외한다.
- 계획서 제목 바로 아래에 `대상 경로`를 적는다. 여러 폴더가 걸리면 주 작업 경로를 기준으로 적는다.
- **현재 상태 분석**: 관련 파일을 실제로 읽고 현재 구조를 정리한다. 추측으로 작성하지 않는다.
- **변경 계획**: 생성/수정/삭제할 파일을 테이블로 정리한다. 사용자가 영향 범위를 한눈에 파악할 수 있어야 한다.
- **구현 상세**: 각 작업 단위별로 구체적인 구현 내용을 기술한다. 메서드 시그니처, 필드 정의, 로직 흐름 등을 포함한다.
- **체크리스트**: 구현 상세의 각 단위에 대응하는 항목을 만든다. 마지막에 `build 검증` 항목을 반드시 포함한다.
- **참고사항**: 참조한 rules, context 문서를 명시한다.

## 5. 체크리스트 완료 처리 규칙

체크리스트 항목이 완료될 때마다 아래 3가지를 모두 적용한다:

1. `- [ ]`를 `- [x]`로 변경한다.
2. 항목 텍스트에 취소선(`~~`)을 적용한다.
3. 완료 시각을 `YYYY-MM-DD HH:mm:ss` 형식으로 기록한다.

### 완료 처리 예시

```markdown
## 체크리스트
- [x] ~~엔티티 작성~~ ✅ 2026-03-17 14:32:10
- [x] ~~리포지토리 작성~~ ✅ 2026-03-17 14:35:22
- [ ] 서비스 작성
- [ ] 컨트롤러 작성
```

## 6. 전체 완료 처리

- 모든 체크리스트가 완료되면:
  1. 계획서 상단에 아래를 추가한다.
  2. `## 구현 결과 요약` 섹션을 작성한다.

```markdown
> ✅ 작업 완료 (YYYY-MM-DD HH:mm:ss)
```

## 7. 주의사항

- 완료 시각은 체크 시점의 실제 현재 시각을 사용한다 (임의 시각 금지).
- 작업 중 계획이 변경되면 체크리스트를 수정하고, 변경 사유를 참고사항에 남긴다.
